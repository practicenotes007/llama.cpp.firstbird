# LLaMA-7B 模型准备完整指南

> 适配项目：`llama.cpp.firstbird`（基于 2023 年 3 月 llama.cpp 早期版本）
> 目标：从零准备模型权重，走通 HF 权重 → GGML 转换 → 量化 → 推理的完整闭环

## 一、前置评估：Gemini 原始文档的问题勘误

以下是对 `LLaMA-7b-模型获取.202303.md`（Gemini 生成）的关键勘误，这些错误会直接导致模型准备流程无法走通：

| # | Gemini 文档的说法 | 实际情况（基于当前项目代码验证） | 严重程度 |
|---|---|---|---|
| 1 | 下载 HuggingFace 的 `pytorch_model-*.bin` 格式权重 | 当前项目的 `convert-pth-to-ggml.py` 读取的是 Meta 原始格式的 `consolidated.0*.pth` 文件，不兼容 HuggingFace 的 `pytorch_model-*.bin` 格式。两种格式的 tensor 命名、分层方式、参数文件结构完全不同 | **致命** |
| 2 | 转换脚本参数为 `params.json`，文件名为 `config.json` | 代码中硬编码读取 `params.json`（不是 `config.json`），HuggingFace 仓库里没有 `params.json` | **致命** |
| 3 | 转换脚本叫 `convert.py` | 实际文件名为 `convert-pth-to-ggml.py` | **中等** |
| 4 | `tokenizer.model` 与 `7B/` 文件夹平级 | 代码路径为 `sys.argv[1] + "/../tokenizer.model"`，即 `tokenizer.model` 需要放在 `7B/` 的**上级**目录。Gemini 说的"平级"恰好是对的，但解释不够清晰——它应该放在 `models/tokenizer.model`，而 `7B/` 在 `models/7B/` | **低** |
| 5 | commit hash `b016a28` 是 2023 年 3 月底的版本 | 当前项目已经是独立维护的 firstbird 版本，不需要 git checkout 回滚 | **低** |
| 6 | 量化命令 `./quantize ... 2` 中 2 代表 Q4_0 | 正确。代码 `quantize.cpp` 中 `itype=2` → `GGML_TYPE_Q4_0`，`itype=3` → `GGML_TYPE_Q4_1` | 正确 |
| 7 | FP16 输出文件约 13GB | 正确。7B 模型 4017 MB 权重 + vocab，FP16 格式约 13GB | 正确 |

**核心结论**：Gemini 文档最大的问题是没有阅读当前项目的实际转换脚本代码，想当然地认为 HuggingFace 的 `.bin` 权重可以直接用。实际上存在两条完全不同的路径，下面分述。

## 二、模型权重格式：两条路径

### 路径 A：Meta 原始格式（`convert-pth-to-ggml.py` 原生支持）✅ 推荐

这是当前项目 `convert-pth-to-ggml.py` **直接支持**的格式。Meta 官方分发的原始权重由以下文件组成：

```
models/
├── 7B/
│   ├── consolidated.00.pth      # 模型权重（PyTorch state_dict，单体文件）
│   └── params.json              # 模型超参数（dim, n_heads, n_layers 等）
└── tokenizer.model              # SentencePiece 分词器
```

**文件来源**：

- `consolidated.00.pth` + `params.json` + `tokenizer.model`：这三个文件是 Meta 在 2023 年 2-3 月通过申请表单分发的原始包。当年 3 月在 4chan 泄露的 torrent 压缩包里也是这个结构。

**当前代码的硬编码路径**（`convert-pth-to-ggml.py` 第 30-31 行）：

```python
fname_hparams   = sys.argv[1] + "/params.json"
fname_tokenizer = sys.argv[1] + "/../tokenizer.model"
```

**`params.json` 的内容格式**（7B 模型）：

```json
{
    "dim": 4096,
    "multiple_of": 256,
    "n_heads": 32,
    "n_layers": 32,
    "vocab_size": 32000
}
```

**`consolidated.00.pth` 的内部结构**：

这是一个标准的 PyTorch `state_dict`，tensor 命名遵循 Meta 的内部命名规范，例如：

```
tok_embeddings.weight
layers.0.attention.wq.weight
layers.0.attention.wk.weight
layers.0.attention.wv.weight
layers.0.attention.wo.weight
layers.0.feed_forward.w1.weight
layers.0.feed_forward.w2.weight
layers.0.feed_forward.w3.weight
layers.0.attention_norm.weight
layers.0.ffn_norm.weight
...
norm.weight
output.weight
```

**7B 模型的分片规则**（代码第 33-42 行）：

```python
def get_n_parts(dim):
    if dim == 4096:   return 1    # 7B:  单文件
    elif dim == 5120: return 2    # 13B: 2 个文件
    elif dim == 6656: return 4    # 33B: 4 个文件
    elif dim == 8192: return 8    # 65B: 8 个文件
```

7B 的 `dim=4096`，所以只需要 1 个 `consolidated.00.pth` 文件，约 12.5 GB。

**如何获取 Meta 原始格式权重**：

1. **官方渠道**：在 https://huggingface.co/meta-llama/Llama-1-7b 申请访问，审核通过后可下载原始格式权重。注意：这个仓库可能不存在或已被移除，Meta 官方组织后来主要维护 Llama 2+。
2. **社区渠道**：在 Hugging Face 上搜索 `consolidated.00.pth` 或 `LLaMA-7B` 原始权重，部分社区成员上传了未经 HF 转换的原始 Meta 格式。
3. **自行从 HF 格式转换**：如果你只能获取 HuggingFace 格式权重，需要先写一个转换脚本将 `.bin` 格式转回 Meta 原始 `.pth` 格式（见路径 B）。

### 路径 B：HuggingFace 格式（需要额外转换步骤）⚠️ 需要额外工作

HuggingFace 仓库 `huggyllama/llama-7b` 提供的是经过 HuggingFace Transformers 库转换后的格式：

```
huggyllama/llama-7b/
├── config.json                    # HF 格式的模型配置（字段名与 params.json 不同）
├── generation_config.json
├── pytorch_model-00001-of-00002.bin  # HF 格式权重，分 2 片
├── pytorch_model-00002-of-00002.bin
├── pytorch_model.bin.index.json      # 分片索引
├── tokenizer.model
├── tokenizer.json
├── tokenizer_config.json
├── special_tokens_map.json
└── model-00001-of-00002.safetensors  # safetensors 格式（与 .bin 内容相同）
    model-00002-of-00002.safetensors
    model.safetensors.index.json
```

**HF 格式与 Meta 原始格式的关键差异**：

| 差异点 | Meta 原始格式 | HuggingFace 格式 |
|--------|-------------|-----------------|
| 文件格式 | `consolidated.00.pth`（单体） | `pytorch_model-00001-of-00002.bin`（分 2 片） |
| 配置文件 | `params.json`（5 个字段） | `config.json`（14+ 字段，字段名不同） |
| tensor 命名 | `layers.0.attention.wq.weight` | `model.layers.0.self_attn.q_proj.weight` |
| 分片方式 | 7B = 1 个文件 | 7B = 2 个文件（按字节均匀切分） |
| 加载方式 | `torch.load("consolidated.00.pth")` | `from_pretrained("huggyllama/llama-7b")` 自动合并分片 |

**如果你只能获取 HF 格式权重，需要一个中间转换脚本**，将 HF 格式转回 Meta 原始格式：

```python
# convert_hf_to_pth.py — 将 HuggingFace 格式权重转换为 Meta 原始格式
# 用法: python3 convert_hf_to_pth.py <hf_model_dir> <output_dir>

import sys
import json
import torch

def convert_hf_to_pth(hf_dir, out_dir):
    from transformers import AutoModelForCausalLM

    model = AutoModelForCausalLM.from_pretrained(hf_dir, torch_dtype=torch.float16)

    state_dict = model.state_dict()

    name_mapping = {
        "model.embed_tokens.weight":                    "tok_embeddings.weight",
        "model.layers.{i}.self_attn.q_proj.weight":     "layers.{i}.attention.wq.weight",
        "model.layers.{i}.self_attn.k_proj.weight":     "layers.{i}.attention.wk.weight",
        "model.layers.{i}.self_attn.v_proj.weight":     "layers.{i}.attention.wv.weight",
        "model.layers.{i}.self_attn.o_proj.weight":     "layers.{i}.attention.wo.weight",
        "model.layers.{i}.mlp.gate_proj.weight":        "layers.{i}.feed_forward.w1.weight",
        "model.layers.{i}.mlp.down_proj.weight":        "layers.{i}.feed_forward.w2.weight",
        "model.layers.{i}.mlp.up_proj.weight":          "layers.{i}.feed_forward.w3.weight",
        "model.layers.{i}.input_layernorm.weight":      "layers.{i}.attention_norm.weight",
        "model.layers.{i}.post_attention_layernorm.weight": "layers.{i}.ffn_norm.weight",
        "model.norm.weight":                            "norm.weight",
        "lm_head.weight":                               "output.weight",
    }

    new_state_dict = {}
    for hf_name, tensor in state_dict.items():
        matched = False
        for hf_pattern, meta_pattern in name_mapping.items():
            if "{i}" in hf_pattern:
                for i in range(32):
                    if hf_name == hf_pattern.format(i=i):
                        new_state_dict[meta_pattern.format(i=i)] = tensor
                        matched = True
                        break
            elif hf_name == hf_pattern:
                new_state_dict[meta_pattern] = tensor
                matched = True
                break
        if not matched:
            print(f"WARNING: unmapped tensor: {hf_name}")

    import os
    os.makedirs(out_dir, exist_ok=True)
    torch.save(new_state_dict, os.path.join(out_dir, "consolidated.00.pth"))

    params = {
        "dim": 4096,
        "multiple_of": 256,
        "n_heads": 32,
        "n_layers": 32,
        "vocab_size": 32000
    }
    with open(os.path.join(out_dir, "params.json"), "w") as f:
        json.dump(params, f, indent=4)

    print(f"Done. Output: {out_dir}/consolidated.00.pth")

if __name__ == "__main__":
    if len(sys.argv) < 3:
        print("Usage: python3 convert_hf_to_pth.py <hf_model_dir> <output_dir>")
        sys.exit(1)
    convert_hf_to_pth(sys.argv[1], sys.argv[2])
```

**GGML 二进制文件结构**（脚本写入顺序）：

```
[4 bytes] 魔数 0x67676d6c ("ggml" in hex)
[4 bytes] vocab_size  = 32000
[4 bytes] dim         = 4096
[4 bytes] multiple_of = 256
[4 bytes] n_heads     = 32
[4 bytes] n_layers    = 32
[4 bytes] rot         = 128 (dim / n_heads, 现已废弃)
[4 bytes] ftype       = 1 (FP16)
--- 词汇表 (32000 个 token) ---
对每个 token:
  [4 bytes] 文本长度
  [N bytes] UTF-8 编码的 token 文本
--- 张量数据 ---
对每个张量:
  [4 bytes] n_dims
  [4 bytes] name_length
  [4 bytes] ftype_cur (0=F32, 1=F16)
  [n_dims * 4 bytes] 各维度大小 (逆序)
  [name_length bytes] 张量名称
  [data bytes] 张量数据 (FP16 或 FP32)
```

**⚠️ 已知问题**：转换脚本中的 tokenizer 解码逻辑有注释标注 "TODO: this is probably wrong"，它用了一种奇怪的解码方式：

```python
text = tokenizer.decode([29889, i]).encode('utf-8')
text = text[1:]  # remove the first byte (it's always '.')
```

这里 `29889` 是 LLaMA tokenizer 中 `.` 的 token ID，解码 `[29889, i]` 后去掉第一个字符 `.`，得到第 `i` 个 token 的文本。这个方法虽然 hack，但在初代 LLaMA 的 SentencePiece tokenizer 上实测可行。

### 步骤 5：量化为 4-bit（Q4_0）

```bash
# Q4_0 量化（itype=2）
./quantize ./models/7B/ggml-model-f16.bin ./models/7B/ggml-model-q4_0.bin 2

# 或 Q4_1 量化（itype=3，精度稍高，文件稍大）
./quantize ./models/7B/ggml-model-f16.bin ./models/7B/ggml-model-q4_1.bin 3
```

**量化类型对比**：

| 类型 | itype | 文件大小 | 精度 | 说明 |
|------|-------|---------|------|------|
| Q4_0 | 2 | ~3.8 GB | 较低 | 每块 32 个权重共享 1 个缩放因子，无偏移 |
| Q4_1 | 3 | ~4.2 GB | 较高 | 每块 32 个权重共享 1 个缩放因子 + 1 个偏移量 |

**量化原理简述**：

- `QK = 32`：每 32 个权重为一个量化块
- Q4_0：每块存储 1 个 FP16 缩放因子 `d` + 16 字节量化数据（每权重 4 bit），反量化公式：`w = d * q`（q 为 -8 到 +7 的整数）
- Q4_1：每块存储 1 个 FP16 缩放因子 `d` + 1 个 FP16 最小值 `m` + 16 字节量化数据，反量化公式：`w = m + d * q`（q 为 0 到 15 的整数）

**`quantize` 命令的验证逻辑**（`quantize.cpp`）：

```cpp
uint32_t magic;
finp.read((char *) &magic, sizeof(magic));
if (magic != 0x67676d6c) {
    fprintf(stderr, "%s: invalid model file (bad magic)\n", __func__);
    return false;
}
```

量化工具会验证输入文件的魔数是否为 `0x67676d6c`（即 "ggml"），确保文件是有效的 GGML 格式。

| `model-*.safetensors` | 13.5 GB | **不需要**（与 .bin 内容重复） |

### 步骤 2：将 HF 格式转换为 Meta 原始格式

```bash
# 安装 transformers 库（仅此步骤需要）
pip install transformers

# 执行转换
python3 convert_hf_to_pth.py ./models/hf-7b ./models/7B
```

其中 `convert_hf_to_pth.py` 脚本见本文第二节"路径 B"部分。它会：

1. 用 `AutoModelForCausalLM.from_pretrained()` 加载 HF 权重（自动合并分片）
2. 将 tensor 命名从 HF 格式映射到 Meta 格式
3. 输出 `consolidated.00.pth` 和 `params.json`

### 步骤 3：复制 tokenizer.model

```bash
cp ./models/hf-7b/tokenizer.model ./models/tokenizer.model
```

### 步骤 4-6：与路径 A 相同

从"步骤 2：建立目录结构"开始，后续步骤完全一致。

## 五、跨服务器迁移与转换（旧服务器已有权重）

如果你的模型权重已经下载到一台低配阿里云服务器（称为 **源服务器 A**），但该服务器内存不足以完成转换，需要新购一台高配服务器（称为 **转换服务器 B**）来执行转换工作。完整的操作流程如下：

### 5.1 架构总览

```
┌──────────────────┐    scp / rsync    ┌──────────────────┐    scp 回传    ┌──────────────┐
│  源服务器 A       │ ───────────────→ │  转换服务器 B     │ ────────────→ │  本地机器     │
│  (低配, 有权重)   │   13.5 GB        │  (高配, 执行转换) │   3.8 GB      │  (运行推理)   │
│  内存 < 4 GB     │                   │  内存 ≥ 16 GB    │               │  任意配置     │
└──────────────────┘                   └──────────────────┘               └──────────────┘
```

### 5.2 转换服务器 B 的购买配置

| 配置项 | 推荐值 | 说明 |
|--------|--------|------|
| 规格 | **ecs.g8i.2xlarge** | 8 vCPU / 32 GB 内存，最省心 |
|  | 或 **ecs.g8i.xlarge** | 4 vCPU / 16 GB 内存，需低内存脚本 |
| 操作系统 | **Ubuntu 22.04 LTS x86_64** | 见下方操作系统选型说明 |
| 系统盘 | 100 GB ESSD Entry | 峰值磁盘占用约 43 GB |
| 地域 | **与源服务器 A 同地域** | 阿里云内网传输免费且速度极快（1 Gbps+） |
| VPC | **与源服务器 A 同 VPC** | 内网互通的前提 |
| 带宽 | 不需要公网带宽 | 内网传输不经过公网 |
| 安全组 | 放行 SSH（22 端口）入方向 | 用于从 A scp 传输和本地远程操作 |

#### 操作系统选型：为什么是 Ubuntu 22.04 LTS

| 考量维度 | Ubuntu 22.04 LTS ✅ | Alibaba Cloud Linux 3 | Ubuntu 24.04 LTS | Rocky Linux 9 |
|---------|---------------------|----------------------|------------------|---------------|
| PyTorch wheel 兼容性 | **最佳**（PyTorch 官方首推平台） | 可用但非首选测试平台 | 较新，旧版 torch 可能有 glibc 兼容问题 | RHEL 系，需额外配置 |
| `build-essential` | `apt install build-essential` 一键搞定 | `yum groupinstall` 可用 | 同 22.04 | 同 ACL |
| Intel g8i 官方推荐 | ✅ 在官方推荐列表 | ✅ 在官方推荐列表 | ❌ **不在** g8i 官方推荐列表 | ✅ 在官方推荐列表 |
| llama.cpp 社区排错资源 | **最多**（几乎所有教程基于 Ubuntu） | 较少 | 较新，坑可能多 | 较少 |
| 空载内存 | ~200 MB | ~150 MB | ~250 MB | ~180 MB |
| LTS 维护到 | 2027 年 | 2029 年 | 2029 年 | 2032 年 |

**不选 Ubuntu 24.04 的原因**：g8i 官方声明支持的操作系统列表中不包含 24.04；且 24.04 的 glibc 版本（2.39）高于 PyTorch 预编译 wheel 的目标版本（2.35），安装旧版 torch 时可能出现兼容问题。

**不选 Alibaba Cloud Linux 3 的原因**：ACL3 空载更省内存（~150 MB vs ~200 MB），但你的场景是临时租用转换服务器，核心诉求是**转换顺利走通**而非长期运行。Ubuntu 在 PyTorch/llama.cpp 社区的排错资源远多于 ACL3，多出的 50 MB 对 32 GB 机器可忽略不计。

**购买页面选择路径**：公共镜像 → Ubuntu → 22.04 64位

> **关键**：购买时务必选择与源服务器 A **同一地域**（如都是华东1-杭州）和 **同一 VPC**，这样才能通过阿里云内网传输，速度可达 1 Gbps（约 125 MB/s），13.5 GB 文件约 2 分钟传完。如果跨地域或跨 VPC，走公网传输会慢很多。

### 5.3 远程拷贝方案

#### 方案一：阿里云内网 SCP（推荐，同地域同 VPC）

**前提**：两台服务器在同一地域、同一 VPC 下，安全组允许内网互通。

在 **转换服务器 B** 上执行（从 B 拉取 A 的文件）：

```bash
# 查看源服务器 A 的内网 IP（在阿里云控制台 → 实例 → 私网 IP）
# 假设为 172.16.0.100

# 在 B 上创建目录
mkdir -p ~/models/hf-7b

# 从 A 拉取整个模型目录
scp -r root@172.16.0.100:/path/to/models/hf-7b/ ~/models/hf-7b/

# 或者只拉取需要的文件（如果 A 上有其他多余文件）
scp root@172.16.0.100:/path/to/models/hf-7b/tokenizer.model ~/models/hf-7b/
scp root@172.16.0.100:/path/to/models/hf-7b/config.json ~/models/hf-7b/
scp root@172.16.0.100:/path/to/models/hf-7b/pytorch_model-00001-of-00002.bin ~/models/hf-7b/
scp root@172.16.0.100:/path/to/models/hf-7b/pytorch_model-00002-of-00002.bin ~/models/hf-7b/
scp root@172.16.0.100:/path/to/models/hf-7b/pytorch_model.bin.index.json ~/models/hf-7b/
```

**传输速度**：阿里云内网约 1 Gbps，13.5 GB 约 **2-3 分钟**。

#### 方案二：rsync 断点续传（推荐，网络不稳定时）

scp 不支持断点续传，如果传输中断需要从头开始。rsync 支持断点续传：

```bash
# 在转换服务器 B 上执行
rsync -avP root@172.16.0.100:/path/to/models/hf-7b/ ~/models/hf-7b/

# -a: 保留文件属性
# -v: 显示详细信息
# -P: 显示进度 + 支持断点续传
```

如果中途中断（如 SSH 超时），再次执行相同命令即可续传。

#### 方案三：跨地域传输（不同地域）

如果两台服务器不在同一地域，内网不通，需要走公网或云企业网：

**选项 A：公网 SCP（慢，但简单）**

```bash
# 在转换服务器 B 上执行，使用源服务器 A 的公网 IP
scp -r root@<A的公网IP>:/path/to/models/hf-7b/ ~/models/hf-7b/

# 如果 A 没有公网带宽，可以在 A 上主动推送到 B
# 在源服务器 A 上执行：
scp -r /path/to/models/hf-7b/ root@<B的公网IP>:~/models/hf-7b/
```

公网传输速度取决于带宽，5 Mbps 带宽下 13.5 GB 约需 **6 小时**。

**选项 B：OSS 中转（推荐，跨地域时最快）**

```bash
# 1. 在源服务器 A 上，上传到阿里云 OSS
#    先在阿里云控制台创建一个 OSS Bucket（与 A 同地域）
pip install oss2
ossutil cp -r /path/to/models/hf-7b/ oss://your-bucket/hf-7b/ --update

# 2. 在转换服务器 B 上，从 OSS 下载
#    如果 B 与 OSS Bucket 不同地域，走公网；同地域走内网
ossutil cp -r oss://your-bucket/hf-7b/ ~/models/hf-7b/ --update

# 3. 完成后清理 OSS
ossutil rm oss://your-bucket/hf-7b/ -r -f
```

OSS 内网传输免费，跨地域走公网按流量计费（约 0.5 元/GB，13.5 GB 约 7 元）。

### 5.4 转换服务器 B 上的完整操作流程

传输完成后，在转换服务器 B 上执行：

```bash
# 0. 安装依赖（项目已提供 requirements.txt）
#    32 GB 内存：一键安装全部

#    16 GB 内存（低内存方案）：core + safetensors

# 如果选择低内存方案，先下载 safetensors 格式到源服务器 A，
# 然后同上 scp 到 B，再执行低内存转换
# pip install transformers  # 仅 from_pretrained() 方案需要

cd ~/llama.cpp.firstbird

# 2. 确认模型文件已到位

# 应看到: config.json  tokenizer.model  pytorch_model-*.bin  ...

# 3. 建立 firstbird 所需的目录结构

cp ~/models/hf-7b/tokenizer.model ~/models/tokenizer.model

# 4. HF → Meta .pth 转换
#    32 GB 内存：直接用 from_pretrained()

#    16 GB 内存：用低内存版

#    <16 GB 内存：用低内存版
# python3 convert-pth-to-ggml-lowmem.py ~/models/7B 1
```

### 5.5 将 Q4_0 量化模型回传到源服务器 A（或本地）

转换完成后，最终产物只有一个 3.8 GB 的 `ggml-model-q4_0.bin` 文件：


## 六、云服务器配置指南（从零开始路径 B）

如果模型权重尚未下载到任何服务器，需要从零开始准备，以下是完整的配置建议。

### 6.1 各步骤资源需求

| 步骤 | 操作 | 峰值内存 | 磁盘占用（累计） | 耗时估算 |
|------|------|---------|----------------|---------|
| 1. 下载 HF 权重 | 网络下载 | ~100 MB | 13.5 GB | 取决于带宽 |
| 2. HF → Meta .pth | `from_pretrained()` 全量加载 | **~20 GB** | 26 GB | 5-10 分钟 |
| 3. .pth → GGML FP16 | `torch.load()` 全量加载 | **~16 GB** | 39 GB | 10-15 分钟 |
| 4. 量化 FP16 → Q4_0 | 读写 GGML 文件 | **~16 GB** | 42.8 GB | 5-10 分钟 |
| 5. 推理运行 | 加载 Q4_0 + KV Cache | **~5 GB** | 3.8 GB | 实时 |

**核心瓶颈**：步骤 2-4 每一步都需要将 13-14 GB 的模型权重完整加载到内存，加上 PyTorch/Python 运行时开销，峰值内存约 16-20 GB。

### 6.2 阿里云 ECS 规格推荐

#### 方案一：推荐配置（最省心）

| 配置项 | 推荐值 | 说明 |
|--------|--------|------|
| 规格 | **ecs.g8i.2xlarge** | 8 vCPU / 32 GB 内存 |
| 操作系统 | **Ubuntu 22.04 LTS x86_64** | PyTorch 兼容性最佳，见 5.2 节操作系统选型说明 |
| 系统盘 | 100 GB ESSD Entry | 峰值需 ~43 GB，留有余量 |
| CPU | Intel Xeon (Cascade Lake) | 8 核加速转换 |
| GPU | 无需 | 纯 CPU 操作 |
| 带宽 | 按量付费 5-10 Mbps | 下载 13.5 GB 约需 3-6 小时 |
| 预估费用 | ~3-5 元/小时 | 全流程约 4-8 小时，总费用约 **15-40 元** |

#### 方案二：最低配置（省钱但需技巧）

| 配置项 | 推荐值 | 说明 |
|--------|--------|------|
| 规格 | **ecs.g8i.xlarge** | 4 vCPU / 16 GB 内存 |
| 操作系统 | **Ubuntu 22.04 LTS x86_64** | 同上 |

#### 方案三：极省配置（safetensors 懒加载）

| 配置项 | 推荐值 | 说明 |
|--------|--------|------|
| 规格 | **ecs.g8i.large** | 2 vCPU / 8 GB 内存 |

**原理**：使用 safetensors 格式（支持逐张量懒加载）替代 PyTorch `.bin` 格式，将步骤 2 的峰值内存从 ~20 GB 降到 **~3 GB**。详见 6.3 节。

### 6.3 低内存转换方案（适用于 ≤16 GB 内存）

当服务器内存不足 20 GB 时，不能使用 `from_pretrained()` 一次性加载整个模型。改用 safetensors 格式 + 逐张量读取，将峰值内存控制在 3 GB 以内。

**下载时选择 safetensors 格式替代 .bin**：

```bash
pip install huggingface_hub safetensors

huggingface-cli download huggyllama/llama-7b \
    tokenizer.model \
    config.json \
    model-00001-of-00002.safetensors \
    model-00002-of-00002.safetensors \
    model.safetensors.index.json \
    --local-dir ./models/hf-7b
```

**低内存转换脚本** `convert_hf_to_pth_lowmem.py`：

```python
# convert_hf_to_pth_lowmem.py
# 逐张量读取 safetensors，峰值内存 ~3 GB（而非 ~20 GB）
# 用法: python3 convert_hf_to_pth_lowmem.py <hf_model_dir> <output_dir>

import sys
import os
import json
import torch
from safetensors import safe_open

HF_TO_META = {
    "model.embed_tokens.weight":                        "tok_embeddings.weight",
    "model.norm.weight":                                "norm.weight",
    "lm_head.weight":                                   "output.weight",
}

LAYER_MAPPING = {
    "self_attn.q_proj.weight":   "attention.wq.weight",
    "self_attn.k_proj.weight":   "attention.wk.weight",
    "self_attn.v_proj.weight":   "attention.wv.weight",
    "self_attn.o_proj.weight":   "attention.wo.weight",
    "mlp.gate_proj.weight":      "feed_forward.w1.weight",
    "mlp.down_proj.weight":      "feed_forward.w2.weight",
    "mlp.up_proj.weight":        "feed_forward.w3.weight",
    "input_layernorm.weight":    "attention_norm.weight",
    "post_attention_layernorm.weight": "ffn_norm.weight",
}

def map_name(hf_name):
    if hf_name in HF_TO_META:
        return HF_TO_META[hf_name]
    if hf_name.startswith("model.layers."):
        parts = hf_name.split(".")
        layer_idx = parts[2]
        suffix = ".".join(parts[3:])
        if suffix in LAYER_MAPPING:
            return f"layers.{layer_idx}.{LAYER_MAPPING[suffix]}"
    print(f"WARNING: unmapped tensor: {hf_name}")
    return None

def convert_lowmem(hf_dir, out_dir):
    os.makedirs(out_dir, exist_ok=True)

    index_path = os.path.join(hf_dir, "model.safetensors.index.json")
    if os.path.exists(index_path):
        with open(index_path, "r") as f:
            index = json.load(f)
        weight_map = index.get("weight_map", {})
        shard_files = set(weight_map.values())
    else:
        shard_files = [f for f in os.listdir(hf_dir) if f.endswith(".safetensors")]
        weight_map = None

    meta_state_dict = {}

    for shard_file in sorted(shard_files):
        shard_path = os.path.join(hf_dir, shard_file)
        print(f"Reading shard: {shard_file}")

        with safe_open(shard_path, framework="pt", device="cpu") as f:
            for hf_name in f.keys():
                meta_name = map_name(hf_name)
                if meta_name is None:
                    continue
                tensor = f.get_tensor(hf_name).contiguous()
                meta_state_dict[meta_name] = tensor

    pth_path = os.path.join(out_dir, "consolidated.00.pth")
    print(f"Saving to {pth_path} ...")
    torch.save(meta_state_dict, pth_path)
    del meta_state_dict

    params = {
        "dim": 4096,
        "multiple_of": 256,
        "n_heads": 32,
        "n_layers": 32,
        "vocab_size": 32000
    }
    params_path = os.path.join(out_dir, "params.json")
    with open(params_path, "w") as f:
        json.dump(params, f, indent=4)

    print(f"Done. Output: {pth_path}")

if __name__ == "__main__":
    if len(sys.argv) < 3:
        print("Usage: python3 convert_hf_to_pth_lowmem.py <hf_model_dir> <output_dir>")
        sys.exit(1)
    convert_lowmem(sys.argv[1], sys.argv[2])
```

**低内存下的完整操作流程**（适用于 8-16 GB 内存）：

```bash
# 0. 安装依赖（低内存版需要 safetensors）
pip install numpy torch sentencepiece safetensors huggingface_hub

# 1. 下载 safetensors 格式权重
huggingface-cli download huggyllama/llama-7b \
    tokenizer.model config.json \
    model-00001-of-00002.safetensors \
    model-00002-of-00002.safetensors \
    model.safetensors.index.json \
    --local-dir ./models/hf-7b

# 2. 低内存转换 HF → Meta .pth（峰值 ~3 GB）
python3 convert_hf_to_pth_lowmem.py ./models/hf-7b ./models/7B

# 3. 复制 tokenizer.model
cp ./models/hf-7b/tokenizer.model ./models/tokenizer.model

# 4. 清理 HF 权重，释放磁盘空间
rm -rf ./models/hf-7b/

# 5. 转换 .pth → GGML FP16（峰值 ~15 GB）
#    ⚠️ 如果内存 <16 GB，此步骤也需要低内存版本
python3 convert-pth-to-ggml.py models/7B 1

# 6. 清理 .pth
rm ./models/7B/consolidated.00.pth

# 7. 量化 FP16 → Q4_0（峰值 ~16 GB，但 GGML 的内存管理更高效）
./quantize ./models/7B/ggml-model-f16.bin ./models/7B/ggml-model-q4_0.bin 2

# 8. 清理 FP16 GGML
rm ./models/7B/ggml-model-f16.bin

# 9. 下载 Q4_0 到本地
# scp ./models/7B/ggml-model-q4_0.bin user@local:/path/to/models/7B/
```

### 6.4 步骤 5-7 的低内存处理

如果内存 < 16 GB，步骤 5（`.pth → GGML FP16`）也会 OOM，因为 `torch.load()` 需要将整个 state_dict 加载到内存。解决方案：使用 `torch.load` 的 `mmap` 模式（PyTorch 2.1+），或改写转换脚本逐张量处理：

```python
# convert-pth-to-ggml-lowmem.py
# 逐张量加载 .pth 文件，峰值内存 ~3 GB
# 用法: python3 convert-pth-to-ggml-lowmem.py models/7B 1

import sys, json, struct, os
import numpy as np
import torch
from sentencepiece import SentencePieceProcessor

dir_model = sys.argv[1]
ftype = int(sys.argv[2]) if len(sys.argv) > 2 else 1
ftype_str = ["f32", "f16"]

fname_hparams   = dir_model + "/params.json"
fname_tokenizer = dir_model + "/../tokenizer.model"
fname_out       = dir_model + "/ggml-model-" + ftype_str[ftype] + ".bin"

with open(fname_hparams, "r") as f:
    hparams = json.load(f)

tokenizer = SentencePieceProcessor(fname_tokenizer)
hparams.update({"vocab_size": tokenizer.vocab_size()})

fout = open(fname_out, "wb")

fout.write(struct.pack("i", 0x67676d6c))
fout.write(struct.pack("i", hparams["vocab_size"]))
fout.write(struct.pack("i", hparams["dim"]))
fout.write(struct.pack("i", hparams["multiple_of"]))
fout.write(struct.pack("i", hparams["n_heads"]))
fout.write(struct.pack("i", hparams["n_layers"]))
fout.write(struct.pack("i", hparams["dim"] // hparams["n_heads"]))
fout.write(struct.pack("i", ftype))

for i in range(32000):
    text = tokenizer.decode([29889, i]).encode('utf-8')
    text = text[1:]
    fout.write(struct.pack("i", len(text)))
    fout.write(text)

# 使用 mmap 模式逐张量读取，不一次性加载整个文件
pth_path = dir_model + "/consolidated.00.pth"
state_dict = torch.load(pth_path, map_location="cpu", mmap=True, weights_only=False)

for name, tensor in state_dict.items():
    if name[-5:] == "freqs":
        continue

    data = tensor.numpy().squeeze()
    n_dims = len(data.shape)

    ftype_cur = 1
    if ftype == 0 or n_dims == 1:
        data = data.astype(np.float32)
        ftype_cur = 0

    sname = name.encode('utf-8')
    fout.write(struct.pack("iii", n_dims, len(sname), ftype_cur))
    for i in range(n_dims):
        fout.write(struct.pack("i", data.shape[n_dims - 1 - i]))
    fout.write(sname)
    data.tofile(fout)

    print(f"  {name}: {list(data.shape)}")

fout.close()
print("Done: " + fname_out)
```

> **注意**：`torch.load(..., mmap=True)` 需要 PyTorch 2.1+。如果版本较低，可升级：`pip install torch --upgrade`。

## 七、Runtime 流程详解

当你执行 `./main -m ./models/7B/ggml-model-q4_0.bin -p "..."` 时，程序内部的完整执行流程如下：

### 7.1 模型加载 (`llama_model_load`)

`main.cpp` 中的 `llama_model_load` 函数执行以下操作：

1. **打开二进制文件**：`std::ifstream(fname, std::ios::binary)`
2. **读取并验证魔数**：必须是 `0x67676d6c`
3. **读取超参数**：`n_vocab`, `n_embd`, `n_mult`, `n_head`, `n_layer`, `n_rot`, `f16`
4. **读取词汇表**：32000 个 token 的文本
5. **分配 GGML 上下文**：`ggml_init()`，预分配足够的内存
6. **逐个读取张量**：对每个张量，读取 `n_dims`、名称、维度、数据类型和原始数据
7. **分配 KV Cache**：`memory_k` 和 `memory_v`，大小为 `n_layer * n_ctx * n_embd`
8. **打印加载信息**：包括模型大小、张量数量、内存占用

### 7.2 Tokenizer 编码 (`llama_tokenize`)

`utils.cpp` 中的 `llama_tokenize` 函数：

1. 使用 SentencePiece 的 BPE 算法将输入 prompt 切分为 token
2. 在开头添加 BOS token（ID=1）
3. 返回 token ID 列表

### 7.3 推理循环 (`llama_eval`)

这是核心循环，每生成一个 token 执行一次：

```
输入 token_ids → Embedding 查表 → 32 层 Transformer → Logits → 采样 → 下一个 token
```

**每层 Transformer 的计算图**（在 `llama_eval` 中用 GGML 操作构建）：

```
输入 x (n_embd)
  │
  ├─→ RMSNorm(attention_norm) ──→ Q = x @ wq  (n_embd × n_embd)
  │                              → K = x @ wk  (n_embd × n_embd)
  │                              → V = x @ wv  (n_embd × n_embd)
  │
  ├─→ RoPE 旋转位置编码 (Q, K)
  │
  ├─→ Self-Attention:
  │     Q @ K^T / sqrt(d_k) → softmax → @ V
  │     ↓
  │     output = attn_result @ wo
  │
  ├─→ 残差连接: x = x + output
  │
  ├─→ RMSNorm(ffn_norm)
  │
  ├─→ FFN (SwiGLU):
  │     w1_out = x @ w1     (gate projection)
  │     w3_out = x @ w3     (up projection)
  │     w2_out = SiLU(w1_out) ⊙ w3_out @ w2  (down projection)
  │
  └─→ 残差连接: x = x + w2_out
```

**GGML 计算图执行**：`ggml_graph_compute()` 遍历计算图的所有节点，根据 `op` 类型分发到对应的计算函数。在 Q4_0 模型中，矩阵乘法的执行路径为：

```
ggml_mul_mat_q4_0()
  → 量化权重反量化为 FP32（块大小 32）
  → 调用 SIMD 加速的向量点积
    → x86: AVX2 / AVX / SSE3 路径
    → ARM: NEON 路径
    → 回退: 标量 C 代码
```

### 7.4 采样与输出

从最后一层的 logits 中采样下一个 token：

1. **Temperature**：`logits /= temperature`
2. **Top-K**：只保留概率最高的 K 个 token
3. **Top-P**：从 Top-K 结果中，只保留累计概率 ≤ P 的 token
4. **随机采样**：按归一化后的概率分布随机选择一个 token
5. **输出**：将 token ID 解码为文本，打印
6. **循环**：将新 token 加入输入，继续生成

## 八、常见问题排查

### Q1: `convert-pth-to-ggml.py` 报错 `FileNotFoundError: params.json`

**原因**：目录中缺少 `params.json` 文件。HuggingFace 仓库提供的是 `config.json`，不是 `params.json`。

**解决**：手动创建 `params.json`（内容见步骤 1）。

### Q2: `convert-pth-to-ggml.py` 报错 `FileNotFoundError: consolidated.00.pth`

**原因**：你下载的是 HuggingFace 的 `pytorch_model-*.bin` 格式，不是 Meta 原始的 `consolidated.00.pth`。

**解决**：使用路径 B 的 `convert_hf_to_pth.py` 先转换格式。

### Q3: `quantize` 报错 `invalid model file (bad magic)`

**原因**：输入文件不是有效的 GGML 格式。可能原因：
- 转换脚本未成功执行
- 文件路径错误
- 文件被截断或损坏

**解决**：检查 `ggml-model-f16.bin` 文件是否存在且大小正确（约 13 GB）。

### Q4: `main` 报错 `failed to open model file`

**原因**：模型文件路径不正确。

**解决**：使用绝对路径或确认相对路径正确。注意 `utils.h` 中的默认模型路径是 `"models/lamma-7B/ggml-model.bin"`（有一个 typo "lamma"，且路径与实际不同），必须通过 `-m` 参数显式指定。

### Q5: 生成结果是乱码

**可能原因**：
- 使用了 `apply_chat_template`：初代 LLaMA 7B 是纯基座模型，不支持对话格式
- 量化精度损失过大：尝试使用 Q4_1 或 FP16
- tokenizer 解码问题：转换脚本中的 tokenizer 解码逻辑有 "TODO: this is probably wrong" 的注释
